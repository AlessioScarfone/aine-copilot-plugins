Spec-Driven Development

mini-sdd

Write the spec. Ship the feature. Never lose the thread.
2026.05.04
v1.0

Most AI-assisted coding sessions start strong and drift. mini-sdd fixes that with one rule: specs before code. Before any feature gets built, it gets written down — what it does, who it's for, and how you'll know it's done. Then Copilot follows the plan, task by task, without losing context.

Step 1
Set the stage
/mini-sdd-context

Tell Copilot what your project is about — its purpose, how it's built, and what it uses. This becomes the shared memory every other step reads from. Do it once; refresh it when things change.

Step 2
Write the spec
/mini-sdd-spec <feature>

Describe the feature: what it should do, who uses it, and how you'll know it's working. Copilot turns this into an ordered checklist. The spec is the contract — it doesn't change during implementation.

Step 3
Ship it
/mini-sdd-implement <spec>

Copilot works through the checklist, one task at a time, writing real code. When the last task is done, the spec is marked complete and the shared memory is updated — ready for your next feature.

No spec, no code. If you try to build before writing a spec, mini-sdd stops and asks you to write one first. It's a small friction that saves a lot of confusion — for you and for your Agent.
